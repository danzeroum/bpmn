01 · SUMÁRIO EXECUTIVO

A Agent Lane (Handoff 12, A-1..A-7) está entregue e é referência de mercado no que se propôs: agente como artefato versionado com lifecycle e promoção assinada, escala normativa de autonomia 0–5 com regra autonomia→gate que bloqueia promoção, critério de parada honesto validado no grafo, simulação mock determinística registrável no ledger e ponte formal agente→humano por escalação (H18). Nenhum runtime mainstream (LangGraph, CrewAI, AutoGen) tem essa camada de governança.

A lacuna é a camada oposta: o desenho ainda não carrega informação suficiente para virar um agente real. Ferramentas são strings livres sem contrato, o modelo é só um nome, budgets não existem como propriedade, schemas são planos, a aresta delegate não tem contrato de I/O nem simulação — e squad hoje é vocabulário de desenho num plugin de exemplo privado, não uma topologia executável. Modelar agentes é funcional; modelar squads ainda não é.

Fundação pronta (única no mercado)
Governança: versionamento id@semver, ledger, assinatura, autonomia→gate, paradas honestas, escalação governada. É o que os runtimes não têm.
Agente individual — 60% do caminho
Estrutura e controle ✓; faltam contratos (tools, schemas, modelo, budget), evals como evidência e qualquer ponte de execução real.
Squad — o gap estrutural
Sem artefato de squad, sem vínculo persona↔agente, sem topologia de orquestração, sem contrato entre agentes, sem simulação multi-agente.

Exatamente 3 nós (llm / tool / decision) + decoradores (memory, planner, errorBoundary — propriedades, nunca nós). Refs id@semver. Naming AgentO sem claim JSON-LD. Zero imports do ecossistema, pinado por teste de acidez.

03 · ANATOMIA DE UM AGENTE PROFISSIONAL — MATRIZ DE COBERTURA

14 capacidades que um agente de produção exige · o que a biblioteca cobre

Referência: práticas consolidadas de engenharia de agentes em 2026 (LangGraph, OpenAI Agents SDK, A2A/MCP, OTel GenAI) cruzadas com o código real do repo. Placar: 4 atende · 6 parcial · 4 falta.

CAPACIDADE
O QUE UM AGENTE PROFISSIONAL EXIGE
NA BIBLIOTECA HOJE
STATUS
Prompt versionado
System prompt como artefato com versão, diff, aprovação e rollback — nunca string solta no código.
promptRef: "prm:research@2.0.0" aponta para btv:prompt da Biblioteca, com lifecycle e promoção assinada.
ATENDE
Controle de fluxo interno
Loop de raciocínio limitado, roteamento por critério verificável, retry com teto, sem loops infinitos.
3 nós + retry com maxRetries obrigatório, ciclo sem parada = erro, stop só sobre output estruturado (confidence proibido). Limitação: decision é binária (onTrue/onFalse), sem roteamento N-way num nó só.
ATENDE
HITL & escalação
Ponto humano obrigatório proporcional à autonomia; caminho formal de "agente pede ajuda".
Autonomia ≤3 exige btv:gate a jusante (erro de promoção com remediação); boundary de escalação no agentTask + escalationRaisedEntry assinável no ledger (H18).
ATENDE
Versionamento & auditoria
Quem mudou o quê, quando; pin de versão em execução; trilha imutável.
Registry com canais/vigência/bindRun, ledger SHA-256 encadeado, assinatura Ed25519, XES para mining. Referência de mercado — nenhum framework de agentes tem isso.
ATENDE
Identidade & papel
Persona com missão, tom, limites de atuação — vinculada formalmente ao agente que a executa.
btv:persona existe como desenho (plugin de exemplo); AgentWorkflow tem só id/name. Não há vínculo persona ↔ agnt-* — a persona "Agente SDR" do canvas e o workflow são mundos separados.
PARCIAL
Contratos de dados
Schemas tipados de entrada/saída (aninhamento, required, enums) validáveis contra o processo e entre agentes.
SchemaShape = Record<string,string> — plano, tokens de tipo ("string", "boolean"). Mapping do agentTask checa existência de chaves, não compatibilidade de tipo com as variáveis do processo.
PARCIAL
Memória & conhecimento
Memória de conversa + bases de conhecimento (RAG) referenciáveis, com escopo, retenção e governança do conteúdo.
Decorator memory {scope: short|long, expiry} — declara que existe, não o quê. Sem artefato "knowledge base" na Biblioteca, sem refs versionadas de conteúdo, sem memória compartilhada entre agentes.
PARCIAL
Guardrails & políticas
Filtros declarativos de entrada/saída (PII, injeção, conteúdo), budgets como regra, política como artefato aprovável.
Autonomia→gate e paradas honestas cobrem o controle; falta o conteúdo: nenhum guardrail de I/O, node type "política" segue pendência aberta (§1.1), budget aparece só como selo decidido não pintar (DECISIONS) — não é propriedade validável.
PARCIAL
Testes & evals
Golden sets versionados, threshold de aprovação como gate de promoção, regressão automática, LLM-as-judge.
Simulação mock determinística (fixture 10× = trilha idêntica) + sessão como evidência no ledger — excelente base. Mas fixtures são ad-hoc do Studio: não há eval set como artefato, nem "X% de casos passam" como critério de promoção.
PARCIAL
Observabilidade de execução
Traces por run (passos, tokens, custo, latência), comparáveis ao desenho aprovado.
Ledger + XES cobrem o design-time (o processo de desenhar). Não há modelo de telemetria de runs reais do agente (OTel GenAI → trilha), nem replay de execução real contra o grafo aprovado — o pacote replay faz isso para BPMN/XES, não para agentes.
PARCIAL
Ferramentas & permissões
Catálogo de tools com schema de I/O, escopos de permissão, taxonomia de erro, alinhamento MCP — a tool é metade do agente.
usesTool: "browser_search" é string livre + params + timeout. btv:connector é só desenho no plugin de exemplo. Sem catálogo versionado, sem contrato, sem permissões, sem MCP. O gap mais crítico do agente individual.
FALTA
Config de modelo & runtime
Parâmetros (temperature, max tokens), provider/fallback, budget de custo/tokens/tempo/passos como propriedades governadas.
LlmConfig = {model, promptRef, structuredOutput?} — o modelo é um nome. Nada de params, fallback, custo estimado, teto de gasto. Um budget estourado hoje não é modelável, logo não é validável nem simulável.
FALTA
Execução real & deploy
Rodar o que se desenhou: live/shadow mode, checkpoints/retomada, interrupts, deploy para engine ou runtime.
Nada por decisão de v1: Live/Shadow ficou fora (H12 §7); interrupts/checkpoints do LangGraph são ignorados-e-declarados no import; deploy Camunda 8/Flowable segue pendência (§2.1). O AIProvider do copilot é a porta pronta e não usada.
FALTA
Multi-agente (squad)
Topologias de orquestração, contratos entre agentes, contexto compartilhado, simulação do conjunto.
delegate é uma ref com semântica a2a:1.0 e nada mais: resolver injetado devolve boolean, a simulação não atravessa a fronteira, o export LangGraph a descarta. Detalhe na seção 05.
FALTA
04 · AGENTE INDIVIDUAL — OS 5 GAPS QUE IMPEDEM "DESENHO → AGENTE REAL"
G1
Tool sem contrato — o desenho não sabe o que a ferramenta aceita nem devolve
Sem schema de I/O da tool, a validação não pode checar se params batem com a assinatura real, a simulação depende de fixture inventada e o data-mapping {{simMapEx2}} só quebra em runtime. A correção natural respeita as cercas do repo: tool como artefato versionado da Biblioteca (promover o btv:connector de exemplo a produto, com inputSchema/outputSchema/escopos/erros), referenciado por usesTool: "tool:browser-search@1.2.0" — mesmo padrão do promptRef, resolução injetada e degradável.
evidência: ToolConfig em agentflow/src/types.ts (usesTool: string livre) · pendencias §1.1
G2
Modelo e budget não são propriedades governadas
Governar autonomia sem governar gasto é meio guardrail: um agente nível 2 pode custar R$ 0,01 ou R$ 10.000 e o modelo não distingue. Falta LlmConfig completo (params, provider, fallback) e um bloco budget {maxTokens, maxCost, maxWallTime, maxSteps} no AgentWorkflow — validável (budget ausente em autonomia ≥2 = warning), simulável (estouro = parada honesta, mesmo padrão do retry) e visível na ponte de escalação, que hoje cita "budget estourou" como gatilho sem que budget exista.
G3
Schemas planos limitam contratos e mapeamentos
Record<string,string> não expressa objeto aninhado, required, enum ou default — exatamente o que APIs de structured output usam. Sem isso não há validação de compatibilidade agentTask ↔ variáveis do processo, nem delegante ↔ delegado. Um subconjunto honesto de JSON Schema (types + required + enum, declarando o que fica fora, padrão S-FEEL) resolve sem violar a cerca anti-JSON-LD.
G4
Evals não são evidência de promoção
A infraestrutura já existe (simulação determinística + sessão no ledger + evaluateGates); falta o artefato: eval set versionado (N casos: input + fixtures + asserções sobre o output) e a regra "promover para ativa exige eval set X @versão com 100% dos casos passando" — evidência SACM no mesmo molde da cobertura de caminhos do simulador BPMN. É o passo que transforma a simulação de demo em garantia.
G5
Nenhuma ponte de execução — nem live, nem export completo
Decisão consciente da v1, mas é o teto de utilidade: o Studio simula com fixtures e para aí. Dois caminhos complementares, ambos preparados pelo código atual: (a) Live/Shadow Mode via AIProvider (H9) — rodar o grafo com chamadas reais, gated por autonomia + gate, trilha real vs. mock lado a lado; (b) export executável — completar o subset LangGraph (interrupts ⇄ gate/escalação; checkpoints ⇄ sessão) e mapear zeebe:taskDefinition no agentTask para deploy Camunda 8 (o passthrough de extensões estrangeiras já foi entregue).
05 · SQUADS — DO VOCABULÁRIO DE DESENHO AO SQUAD EXECUTÁVEL

Hoje "squad" existe em duas metades que não se tocam: o desenho macro (btv:squad/persona em plugin de exemplo, privado) e o micro-comportamento (AgentWorkflow com aresta delegate). Entre elas não há artefato, contrato nem semântica. Seis lacunas, em ordem de dependência:

S1 · Squad não é artefato
Não existe "SQUAD" na Biblioteca: nada agrega refs de agentes, versões compatíveis e papel de cada um. Sem manifesto, não há promoção coordenada — promover o squad deveria verificar a vigência de todos os membros (a regra de warning por agente já existe; a agregação não).
S2 · Persona ↔ agente sem vínculo
btv:persona carrega papel/nome no desenho; AgentWorkflow carrega o comportamento. Falta personaRef no workflow (ou agentWorkflowRef na persona) — o mesmo padrão id@semver de promptRef. Sem isso o organograma e a execução contam histórias diferentes.
S3 · delegate sem contrato
resolveDelegate?: (ref) => boolean só responde "existe?". Deveria devolver o workflow para validar: outputSchema do chamador cobre o inputSchema do delegado? Há ciclo A→B→A entre agentes? Qual o autonomia efetivo do conjunto (delegar para um nível 5 contamina o chamador?). Tudo isso é validação headless barata — e hoje impossível.
S4 · Sem topologia de orquestração
Supervisor/hierárquico, sequencial (pipeline), paralelo com agregação, handoff — os 4 padrões que todo runtime mainstream de 2026 executa. O corte de voting/debate ("sem runtime mainstream", H12 §0) envelheceu: LangGraph supervisor/swarm e AutoGen são mainstream. A topologia pode ser um campo do manifesto de squad + validação, sem 4º tipo de nó — a cerca §1.3 sobrevive.
S5 · Simulação não atravessa agentes
internalSuccessors descarta arestas delegate: a trilha para na fronteira do agente. Simulação de squad = seguir o delegate com o workflow resolvido injetado + fixtures por agente, paradas honestas cross-agente ("delegado bloqueou em X"), determinismo preservado. Sem isso, squad não é testável antes de existir.
S6 · Sem contexto compartilhado
Cada agente tem memory decorator próprio; um squad precisa declarar o que é compartilhado (blackboard/estado comum) e o que é privado — inclusive para a LGPD do conteúdo. Declaração no manifesto (refs de knowledge + escopo), não implementação de store: modelagem honesta, execução injetada.

No BPM macro, o squad também carece de semântica: a lane "Agentes IA" dos protótipos é convenção visual (sem mapeamento formal squad↔lane/pool), e multi-instância sobre agentTask (N agentes paralelos sobre uma coleção) herda loopCharacteristics genérico sem regra própria de autonomia/gate agregado.

06 · DOIS MODOS DE VIDA — NO FLUXO BPM E INDEPENDENTE
Agente embutido no processo BPM
COBERTO
agentTask com input/outputMapping · boundary de erro proposto (undoável) · boundary de escalação governada · regra autonomia→gate na promoção · warning de vigência · lint · round-trip com degradação graciosa em editor externo.
FALTA
Tipagem das variáveis do processo (o mapping não valida tipos — G3) · deploy para engine (Camunda 8 via zeebe:taskDefinition, pendência §2.1) · semântica de multi-instância para agentes · correlação de mensagens para agente de longa duração (checkpoints).
Agente independente (fora do BPM)
COBERTO
Agente é artefato de primeira classe da Biblioteca (lifecycle completo sem nenhum BPMN em volta) · Studio degrada para JSON local sem registry · export LangGraph do subconjunto.
FALTA
Gatilho de entrada modelado (schedule/webhook/evento — hoje o agente só "existe", não "começa") · manifesto de publicação (agent card A2A: capacidades, schemas, endpoint) · export executável completo (interrupts/checkpoints) · superfície de serviço (o agente como API consumível por outros sistemas).

08 · VEREDITO

A biblioteca resolveu o problema que ninguém mais resolveu — governar agentes (quem aprovou, que versão roda, quanto pode decidir sozinho, quando um humano assina) — e deixou intacto o problema que todos os outros resolvem: fazer o desenho virar comportamento. O caminho não é virar runtime, e sim fechar os contratos que tornam o desenho suficiente: tools com schema (SQ-1), budget governado (SQ-2), schemas compatíveis (SQ-3), delegate com contrato (SQ-4) e o squad como artefato com topologia (SQ-5). Com esses cinco, um export executável e um eval set deixam de ser features e viram consequência.

Tudo acima respeita as cercas vigentes: 3 nós para sempre, capacidades novas como decoradores/propriedades/artefatos, integração por injeção degradável, honestidade declarada ("o que não mapeia é dito, nunca convertido em silêncio"). O placar honesto: agente individual modelável e governável hoje · funcional de ponta a ponta após P0 · squads exigem o Handoff 22 inteiro.
